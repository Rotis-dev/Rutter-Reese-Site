Troubleshooting — Network/Power Failover Triage (Generic Example)

Version: v1.1 — 2026-01-01 · Sanitized portfolio sample — not operational guidance.

Scope and intent

This guide demonstrates a troubleshooting workflow for “telemetry offline / flapping” incidents where the root cause may be network, power, or failover behavior. It’s generic by design.

Symptoms

  • Telemetry offline / stale timestamps
  • Remote commands time out
  • Intermittent connection (“flapping” alarms)

Goal

  • Confirm scope (single site vs multiple)
  • Stabilize and restore safely (least-risk first)
  • Escalate with clean notes if not recoverable

Capture (start)

  • Start time (last known good telemetry)
  • Exact symptoms + any related alarms
  • Recent changes (maintenance, comms/power work)

Fast checks (2 minutes)

  1. Confirm the monitoring platform is healthy (not a dashboard/API outage).
  2. Check scope: is more than one site/device affected?
  3. Record last known good telemetry time (defines the incident window).

Decision tree

  • If multiple sites are affected → treat as platform/network issue; escalate.
  • If only one site is affected → continue local triage.
  • If last update < 5 minutes ago → wait 5 minutes and re-check (avoid thrash).
  • If last update > 10 minutes ago → proceed to recovery steps (if permitted).
Heuristic: “network vs power vs failover”
  • Network pattern: comms alarms, stale telemetry, but power indicators seem normal.
  • Power pattern: sudden drop + multiple subsystems offline simultaneously; repeated resets.
  • Failover pattern: intermittent connectivity at regular intervals (switching paths).

Recovery steps (least-risk first)

  1. Re-poll / refresh telemetry and confirm timestamps are truly stale (not UI delay).
  2. Check comms status indicators (generic): gateway online, tunnel up, heartbeat present.
  3. If allowed: perform a single least-disruptive recovery action (example: restart comms service) and monitor for 10 minutes.
  4. If still offline: stop repeated actions; prepare escalation package and request field verification.
Escalation package (copy/paste)
  • Symptoms + start time (last known good)
  • Scope (single vs multiple sites)
  • Fast checks performed + results
  • Actions attempted (what/when) + outcome
  • Current status + any constraints (maintenance window, DNO, etc.)

Verification

  • Telemetry timestamps advance normally for 10 minutes.
  • No repeated comms alarms during the verification period.
  • Closeout notes entered (what/when/why + follow-ups).
