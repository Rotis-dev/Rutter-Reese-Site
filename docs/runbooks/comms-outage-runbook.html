Runbook — Communications Outage (Template)

Version: v0.2 — 2026-01-01 · Sanitized portfolio sample — not operational guidance.

Scope and intent

This document demonstrates a standard comms-outage runbook structure: define scope, run safe checks, attempt least-risk recovery actions (if authorized), and escalate with clean notes when recovery fails.

Trigger

  • Alarm indicates communications loss (example: “Comms Lost”, “Gateway Offline”).
  • Telemetry is stale beyond threshold (example: no updates for > X minutes).
  • Multiple related comms alarms within 5 minutes.

Goal

  • Determine scope: single site vs fleet vs upstream service.
  • Restore comms safely (only approved actions).
  • Escalate fast with evidence if not recoverable.

Capture (start)

  • Timestamp(s), alarm text, severity.
  • Last known good telemetry time.
  • What changed recently (maintenance, config, weather, ISP notice).

Safety checks (do first)

  1. Confirm you are in the correct environment (site/system) before any action.
  2. Check for “Do Not Operate” / lockout tags / active maintenance windows.
  3. Do not initiate disruptive remote actions unless authorized by procedure.
  4. If there is any safety risk or uncertainty, stop and escalate.

Scope triage (is it local or widespread?)

  1. Single site? Check if only one site/device is affected.
  2. Multiple sites? Check if sites on the same carrier/region are affected.
  3. System-wide? Check ROC/SCADA platform status, gateway services, VPN, and upstream dependencies.
Quick checklist (generic)
  • Any scheduled maintenance or change window?
  • Any recent configuration changes (network/VPN/certs/credentials)?
  • Any external outage advisory (ISP/carrier/platform)?
  • Is power stable at the site (if power loss could cause comms loss)?

Triage steps (least-risk first)

  1. Record the current status (online/offline/stale) and confirm the symptom is real (not a UI caching issue).
  2. Check last telemetry timestamp and whether heartbeats are missing or delayed.
  3. Check dependencies: power state, gateway/service health, VPN/tunnel status, and network path.
  4. Attempt the least-risk permitted action (examples: refresh/re-poll, restart a comms service, switch to a standby path) only if authorized.
  5. Re-check telemetry timestamps and alarm state after each action.

Decision points

  • If telemetry resumes and alarms clear, go to Verification.
  • If comms remain down beyond the threshold (example: > 10 minutes), escalate per matrix.
  • If the outage appears widespread (carrier/platform), prioritize notification + ticketing rather than repeated local actions.
  • If symptoms suggest equipment damage or unsafe conditions, stop remote actions and escalate.
Escalation matrix (generic)
  • Critical: immediate call to on-call engineer + field dispatch if required.
  • High: notify on-call engineer; dispatch if not recovered within threshold.
  • Moderate: create ticket + monitor; notify during business hours.

Escalation package (what to send)

  • Symptoms: “telemetry stale since …” / “gateway offline” / “VPN down” (generic).
  • Scope: which sites/devices impacted (count + pattern).
  • Checks performed + results (ordered list).
  • Actions attempted (exactly what, when) + outcome.
  • Any safety or operational constraints (maintenance window, DNO, etc.).

Verification

  1. Telemetry updates are current (timestamps advancing normally) for at least 10 minutes.
  2. No new related comms alarms; prior alarms are cleared or properly acknowledged.
  3. Update ticket/log with actions taken, recovery time, and remaining follow-ups.

Closeout notes (recordkeeping)

  • What happened (1–2 sentences)
  • What you did (bullets, ordered)
  • Who you notified
  • What still needs follow-up
