Sanitization Checklist

This page is the “publish gate” for anything work-derived. When in doubt, remove it or rewrite it as a generic scenario.

Goal: demonstrate structure + clarity without exposing company identity, sensitive operations, or security details.

Rule 0 — Permission and ownership

  • Assume you cannot publish internal documentation unless you have explicit permission or your employer policy allows it.
  • If your employer policy forbids sharing work artifacts (even sanitized), do not publish—use original samples instead.
  • Employers typically own work produced during employment; sharing it publicly can still be a problem even if you wrote it.
Safe alternatives
  • Write original “generic but realistic” samples (like the docs on this site).
  • Create case studies describing your approach and outcomes without revealing proprietary details.
  • Contribute documentation to an open-source project (public and verifiable).

Never publish (hard rules)

  • Credentials, secrets, API keys, tokens, certificates, VPN configs, private URLs, internal hostnames.
  • Network diagrams/topology that reveal real architecture, IP ranges, firewall rules, or access paths.
  • Customer lists, site names, asset IDs, ticket IDs, internal incident timelines tied to a real org.
  • Screenshots from proprietary tools/dashboards (SCADA, internal monitoring, ticket systems), unless fully mocked.
  • Security procedures that could enable abuse (bypass methods, “break glass” instructions, privileged workflows).
  • Personal data (PII) for anyone: names, emails, phone numbers, addresses, employee IDs, etc.

Sanitization steps (before you publish)

1) Remove identifiers

  • Company/customer names, sites, regions, asset IDs
  • Vendor/integrator identifiers if they narrow identity
  • Ticket/incident numbers, internal team names

2) Generalize parameters

  • Thresholds, setpoints, limits, timeouts → use “X minutes” / “within expected range”
  • Precise alarm texts → use examples that don’t match real strings
  • Exact tools/platform names → “monitoring dashboard” / “ticket system”

3) Remove security detail

  • Access paths, roles, permissions
  • Escalation contacts and on-call routing
  • Any “how to exploit” information
PII check (people data)

Treat personal data as sensitive by default. Remove or replace with fictional placeholders.

  • Names, email addresses, phone numbers
  • Employee IDs / badge numbers
  • Locations more specific than necessary
  • Any combination that could re-identify a person
Screenshots and metadata
  • Prefer recreated mock screenshots or diagrams.
  • If you must use an image: blur/remove all identifiers, time ranges, URLs, and internal labels.
  • Check file metadata (filenames, author, comments) before publishing.
“Real doc” → “portfolio doc” conversion
  1. Keep the structure (trigger, safety checks, steps, decisions, verification).
  2. Rewrite the scenario to be generic but realistic.
  3. Replace system names with neutral placeholders (Subsystem A, Site 01, Monitoring Dashboard).
  4. Remove unique operational fingerprints (exact alarm strings, exact time windows, unique constraints).
  5. Add a disclaimer header: “Sanitized portfolio sample — not operational guidance.”

Pre-publish QA (fast pass)

  1. Search the page for company/site/vendor names and remove them.
  2. Search for: @, http, vpn, token, key, password, cert, ssh and remove anything sensitive.
  3. Confirm there are no screenshots from real internal tools.
  4. Confirm escalation contacts are generic roles (not real names/phones/emails).
  5. Confirm the doc still “works” as a sample: clear steps, decisions, verification.
Repo safety (avoid publishing secrets by accident)

Use tooling and habits that reduce accidental secret exposure in Git repos.

  • Never commit files with credentials/secrets.
  • Consider a secret scanner in your workflow before pushing public changes.

If a hiring manager asks for “real samples”

  • Offer a sanitized version or an original sample written specifically for them.
  • If they need a format match (Confluence/Document360/Zendesk), provide the same content in that structure.
  • If confidentiality is strict, provide a case study of your process and outcomes without files.
Note: This is a practical checklist, not legal advice. When unsure, default to “don’t publish.”
