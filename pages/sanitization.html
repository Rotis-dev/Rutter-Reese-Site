Sanitization Checklist

This is the publish gate for anything derived from real operations. Goal: demonstrate structure + clarity without exposing identity, sensitive operations, or security details.

Permission Never publish Sanitization steps Pre-publish QA Repo safety If asked for “real” samples

Samples on this site are intentionally generic. If in doubt: remove it or rewrite it as a new scenario.

Rule 0 — Permission and ownership

  • Assume you cannot publish internal documentation unless your employer policy explicitly allows it.
  • Even if you wrote it, employers commonly own work created during employment. Publishing may still violate policy or agreements.
  • When unsure, publish an original sample that demonstrates your approach without using internal artifacts.
Safe alternatives
  • Write original “generic but realistic” samples (preferred).
  • Write a case study (problem → approach → result) without files, names, screenshots, or internal tools.
  • Contribute docs to an open-source project (public + verifiable).

Never publish (hard rules)

  • Credentials/secrets: API keys, tokens, passwords, certificates, VPN configs, private URLs, internal hostnames.
  • Network architecture/topology that reveals real access paths, IP ranges, firewall rules, or privileged workflows.
  • Company/customer identifiers: site names, asset IDs, ticket IDs, customer lists, real incident timelines tied to an org.
  • Screenshots from real internal tools/dashboards (SCADA, monitoring, ticket systems) unless fully mocked.
  • Security procedures that could enable abuse (bypass, break-glass, escalation routes, privileged steps).
  • Personal data (PII): names, personal emails, phone numbers, addresses, employee IDs, etc.

If a sample contains PII, treat it as high risk and remove it. NIST guidance emphasizes protecting PII from inappropriate access/use/disclosure. :contentReference[oaicite:0]{index=0}

Sanitization steps (before you publish)

1) Remove identifiers

  • Company/customer names, sites, regions, asset IDs
  • Vendor/integrator identifiers that narrow identity
  • Ticket/incident numbers, team names, on-call labels

2) Generalize parameters

  • Setpoints/thresholds/timeouts → “within expected range” / “X minutes”
  • Exact alarm texts → examples that don’t match real strings
  • Tool names → “monitoring dashboard” / “ticket system”

3) Remove security detail

  • Access paths, role names, permissions
  • Escalation contacts and routing details
  • Anything that could be repurposed for abuse

Convert “real doc” → “portfolio doc”

  1. Keep the structure (trigger, checks, steps, decisions, verification).
  2. Rewrite the scenario to be generic but realistic.
  3. Replace system names with placeholders (Subsystem A, Site 01).
  4. Remove unique fingerprints (exact strings, exact constraints).
  5. Add a disclaimer header: “Sanitized portfolio sample — not operational guidance.”

Redaction/de-identification is a recognized approach for reducing re-identification risk when sharing data. :contentReference[oaicite:1]{index=1}

Pre-publish QA (fast pass)

  1. Search for company/site/vendor names and remove them.
  2. Search for: @, http, vpn, token, key, password, cert, ssh and remove anything sensitive.
  3. Confirm there are no screenshots from real internal tools.
  4. Confirm escalation contacts are generic roles (not real people).
  5. Confirm the doc still works as a sample (clear steps, decisions, verification).
Logging / copy-paste traps
  • Don’t paste real URLs, hostnames, or system identifiers “as examples.”
  • Don’t include real alarm strings or exact query filters.
  • Don’t include secrets in code blocks or configs—OWASP explicitly warns against hardcoding secrets in repos/logs. :contentReference[oaicite:2]{index=2}

Repo safety (avoid accidental leaks)

  • Never commit secrets (even “temporary” ones).
  • Use automated checks where possible. GitHub push protection is designed to block pushes that contain supported secrets. :contentReference[oaicite:3]{index=3}
  • Keep samples generic—no real tool screenshots, URLs, or internal identifiers.

If a hiring manager asks for “real samples”

  • Offer a sanitized version or write a fresh sample tailored to their format.
  • If they want a platform match (Confluence/Document360/Zendesk/Markdown repo), deliver the same content in that structure.
  • If confidentiality is strict, provide a short case study instead of files.

Disclaimer

Practical guidance, not legal advice. When unsure, default to “don’t publish.”
