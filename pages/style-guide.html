Style Guide

Rules that keep documentation consistent, auditable, and fast to use under pressure. Built for runbooks, SOPs, and troubleshooting guides.

Principles Templates Procedures Decision points Warnings Verification Versioning References

This page is part of Process. For writing samples, go to Portfolio.

Principles

Write for reality

Assume the reader is busy, tired, and time-limited. Make the “next action” obvious.

Structure beats prose

Use repeatable sections so every doc is scannable: trigger, checks, steps, decisions, escalation, verification.

Evidence over opinions

Capture timestamps, observed states, actions taken, and outcomes. A good closeout can be audited later.

Document templates

These are the standard shapes used across the site.

Runbook

Incident response: an alarm/incident happens → what to do next.

Runbook skeleton
  1. Title + Version + Date + Disclaimer
  2. Trigger (symptoms / alarm text examples)
  3. Safety checks (do first)
  4. Triage steps (least-risk first)
  5. Decision points (IF/THEN → next action)
  6. Escalation (who/when/how + what to send)
  7. Verification (acceptance criteria)
  8. Closeout notes (what happened / what you did / follow-ups)
  9. Changes (short changelog)

SOP

Routine process: repeatable work with quality checks and records.

SOP skeleton
  1. Purpose + Audience
  2. Scope (included/excluded)
  3. Prerequisites (access/tools/permissions)
  4. Procedure (numbered steps)
  5. Quality checks (pass/fail completeness)
  6. Records (what to log + where)
  7. Changes (short changelog)

Troubleshooting

Symptoms → fast checks → decision tree → recovery → verification.

Troubleshooting skeleton
  1. Symptoms (what the reader sees)
  2. Fast checks (2 minutes; confirm scope)
  3. Decision tree (branching logic)
  4. Recovery steps (least-risk first)
  5. Escalation package (copy/paste)
  6. Verification (clear pass/fail)
  7. Changes (short changelog)

Procedure writing rules

Use imperative steps

Start with a verb: “Open…”, “Verify…”, “Record…”.

One step = one action

Avoid “and” inside steps. Split steps so each action can be checked.

Prefer lists over paragraphs

Sequences use numbered lists. Options use bullets.

Procedure formatting

  • Use numbered steps for actions. Keep step text short.
  • Use consistent names for the same thing (terms, tools, states).
  • Place warnings immediately before the step they affect.
  • Use a “do first” block when ordering matters (safety checks, environment checks).

Example: good steps vs bad steps

Good
  1. Open the incident ticket and record the alarm timestamp.
  2. Verify the correct site/system is selected.
  3. Check whether a maintenance window is active.
  4. Re-check telemetry timestamps after 2 minutes.
Bad (multi-action, unclear)
  1. Go look at the alarms and see if anything is wrong and then try a reset and tell the engineer.

Escalation package (copy/paste block)

Include this in runbooks and troubleshooting guides so escalation is clean. 

SUMMARY (1–2 lines):
SCOPE (single site vs widespread):
START TIME (last known good / first alarm time):
SYMPTOMS (exact alarm text or observed state):
CHECKS PERFORMED (bullets):
ACTIONS ATTEMPTED (bullets; least-risk first):
CURRENT STATUS (what is true right now):
RISKS / SAFETY CONCERNS:
REQUEST (what you want the receiver to do next):

Decision points

Write IF/THEN

Each decision must lead to a specific next action (ideally a named section).

Use thresholds

Time thresholds reduce churn (example: wait 5 minutes before repeating recovery actions).

Escalation is a decision

Escalate when risk increases, recovery fails past threshold, or evidence suggests damage/unsafe state.

Example decision block

  • If telemetry resumes and alarms clear → go to Verification.
  • If telemetry remains stale > 10 minutes → Escalate and send the escalation package.
  • If signs indicate possible equipment damage → stop actions and escalate immediately.

Warnings, cautions, notes

WARNING

Safety risk or equipment damage risk. Short, actionable, placed immediately before the step it affects.

CAUTION

Procedure risk (wrong environment, wrong state, unintended disruption).

NOTE / DISCLAIMER

Helpful context that does not change the step sequence (sanitized sample, scope, authorization limits).

Rule

  • Callouts must be short and actionable.
  • Put callouts immediately before the step they affect.
  • If a warning changes what the reader does, it belongs in steps or decisions.

Verification and records

Verification is pass/fail

Define “done”: stable readings for 10 minutes, alarms cleared, timestamps advancing, etc.

Recordkeeping is part of the job

Closeout notes: what happened, what you did, who you notified, follow-ups.

Escalate with evidence

Escalation packages should be copy/paste: symptoms, scope, checks, actions attempted, current status.

Versioning and review

Version header

Every doc has: version, date, and a short disclaimer (especially for sanitized samples).

Changes block

Keep a short “Changes” section at the bottom of each doc: what changed and why.

Review loop

Prefer small revisions over big rewrites. Changes should be visible and explainable.

Style promise

Consistent headings, action-first steps, explicit decision points, and testable verification criteria.
