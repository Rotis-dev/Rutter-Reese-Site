How to write a good changelog entry

  • Write for the reader (what changed + why it matters).
  • Prefer categories: Added / Changed / Fixed / Security.
  • Keep entries short, consistent, and dated.

Per-document versioning

Each portfolio doc (runbook/SOP/troubleshooting) should carry its own version header plus a small “Changes” block at the bottom.

Header block (top of doc)

Keep it consistent across all docs.

Version: v1.0 — 2026-01-01
Sanitized portfolio sample — not operational guidance.

Changes block (bottom of doc)

2–5 bullets is enough.

Changes
- v1.0: Initial published sample.
- v0.2: Clarified decision points and verification criteria.

Version meaning

Optional semantics if you want stricter rules:

  • MAJOR: structure change that breaks expectations
  • MINOR: adds sections/coverage without breaking
  • PATCH: fixes wording/typos/links

Git tags (optional, but clean)

Tag releases so a “site version” is easy to reference (v0.1, v0.2…).

Example commands

# From your repo folder
git tag -a v0.1 -m "v0.1 initial portfolio site skeleton"
git push origin v0.1

Keep commit messages aligned with changelog bullets.
